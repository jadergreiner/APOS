# Sprint 0.2: Guia de Preparação para Entrevistas JTBD

**Objetivo:** Conduzir 5 entrevistas com foco em descoberta de jobs reais
**Framework:** Jobs to Be Done (JTBD)
**Duração:** 45 min/persona
**Mindset:** Curiosidade + Escuta, não pitch

---

## Mindset Correto (IMPORTANTE)

### ❌ O QUE NÃO FAZER

- Pitch APOS (evite falar de features/produto)
- Fazer perguntas leading ("Concorda que X seria bom?")
- Argumentar se persona discordar
- Falar mais que 30% do tempo
- Tomar decisão baseado em 1 entrevista

### ✅ O QUE FAZER

- Escuta ativa (80% você ouve, 20% você fala)
- Fazer perguntas abertas ("Como você...?", "Conte-me...")
- Seguir curiosidade genuína
- Documentar exatamente o que foi dito (não interpretação)
- Validar com 5+ personas antes de concluir

---

## Estrutura da Entrevista (45 min)

### 1️⃣ **Warm-up (5 min)** — Construir rapport

**Objetivo:** Pessoa se sentir confortável, começar a abrir

**Script:**
```
"Obrigado por estar aqui! Antes de mais nada, obrigado por dedicar 45 min.

Vou fazer perguntas sobre como você trabalha hoje — não há respostas certas. 
Estou aqui para entender sua realidade, não para convencer você de nada.

Posso fazer perguntas no meio se não entender algo?

Tá bom? Vamos lá."
```

**Perguntas de Warm-up:**
- "Conte-me sobre sua função"
- "Qual é o tamanho da sua equipe?"
- "Há quanto tempo você está nesse role?"

⏱️ **Tempo:** Exato 5 min (não deixar passar de 10 min)

---

### 2️⃣ **Status Quo (10 min)** — Entender realidade atual

**Objetivo:** Documentar como pessoa trabalha HOJE (não o ideal)

**Perguntas Principais:**

**P1: Documentação de Contexto (3 min)**
```
"Como você hoje documenta contexto estratégico de uma tarefa ou feature?

Por exemplo, se sua equipe está trabalhando em 'adicionar autenticação SSO',
como você captura 'por que isso é importante agora?'"
```

**Esperado:** Descrição de processo real
- Onde fica documentado (Jira, Notion, Confluence, Docs, email)
- Quem faz (PM, Tech Lead, stakeholder)
- Quando é feito (antes? durante? depois?)
- Por quanto tempo fica atualizado

**P2: Retrabalho de Explicação (4 min)**
```
"Com que frequência você acaba re-explicando POR QUÊ algo está priorizado?

Tipo, alguém da equipe pergunta: 'Por que estamos fazendo isso agora?'
e você precisa re-explicar do zero?"
```

**Esperado:** Frequência + contexto
- "3x por semana com novos membros"
- "Toda reunião de planejamento"
- "Sempre que mudança de prioridade"

**P3: Alinhamento (3 min)**
```
"Como você valida que uma feature entregue mapeia para um OKR real?

Ou seja, você terminou de entregar X — como você sabe se era importante mesmo?"
```

**Esperado:** Processo de validação
- Manual (review com stakeholder)
- Automático (verificação)
- Ad-hoc (só se problema aparecer)

⏱️ **Tempo:** 10 min fixo

---

### 3️⃣ **Jobs Discovery (20 min)** — Explorar a progressão

**Objetivo:** Entender forces que movem pessoa

**Formato:** Conversa natural, mas guiada por jobs framework

**P4: Push Forces — O que EMPURRA para solução (5 min)**

```
"Se a situação é X (como você descreveu), o que isso causa pra você?

Como você se sente quando precisa re-explicar contexto 3x por semana?
Qual é o impacto (tempo, frustração, qualidade)?"
```

**Listen for:**
- Frustração ("Cansa explicar")
- Cost ("Perde 5h/semana em alinhamento")
- Risk ("Features de-priorizadas aparecem 2 meses depois")
- Opportunity cost ("Time poderia estar coding")

**P5: Pull Forces — O que PUXA para APOS (7 min)**

```
"Se existisse uma ferramenta que automaticamente conectasse 
Tarefa → Feature → OKR → Métrica em um dashboard...

Como você usaria? O que mudaria no seu workflow?"
```

**Listen for:**
- Visualização ("Ver Task→OKR em 10s")
- Automação ("Não ter que fazer roll-up manual")
- Confiança ("Saber que delivery mapeia para estratégia")
- Velocidade ("-80% tempo em alinhamento")

**P6: Habit Forces — O que mantém no STATUS QUO (4 min)**

```
"Se tal ferramenta existisse... o que dificultaria você adotar?

O que você gostaria de manter do seu workflow atual?
Qual mudança seria muito disruptiva?"
```

**Listen for:**
- Switching cost ("Jira tá integrado em tudo")
- Familiarity ("Conheco Jira de cor")
- Effort ("Curva de aprendizado")
- Inertia ("'Se não tá quebrado...'")

**P7: Anxiety Forces — O que gera MEDO (4 min)**

```
"O que geraria preocupação em adotar uma ferramenta nova?

Quais são seus piores cenários?"
```

**Listen for:**
- Risk ("E se não funcionar?")
- Data quality ("Como garante que dados estão certos?")
- Integration ("Como se integra com nosso stack?")
- Support ("E se quebrar? Quem suporta?")

⏱️ **Tempo:** 20 min fixo

---

### 4️⃣ **Close (10 min)** — Próximos Passos

**P8: Síntese (5 min)**

```
"Deixa eu ver se entendi direito...

[Resumir 3-5 pontos principais que você ouviu]

Tá certo?"
```

**Objetivo:** Validar que você entendeu (não mal-interpretar)

**P9: Interesse (3 min)**

```
"Você estaria aberto a ser early adopter de APOS?
Tipo, testar quando tiver MVP disponível (provavelmente Q3/Q4 2026)?"
```

**Esperado:** Sim/Não + condições

**P10: Referência (2 min)**

```
"Conhece outra pessoa que seria bom eu entrevistar?
Alguém com pain similar?"
```

**Esperado:** Nomes + contexto (para chain sampling)

⏱️ **Tempo:** 10 min (máximo 50 min total)

## Técnicas de Entrevista (Pratique Antes!)

### 1. **Escuta Ativa**

**Fazer:**
- Nodding (mesmo que virtual — "sim, entendi")
- Notas breves (não escrever tudo)
- Eye contact (se vídeo)
- Pausas (deixar pessoa terminar completamente)

**NÃO fazer:**
- Interrupting
- Checando email/chat
- Pensando na próxima pergunta (listen first)

### 2. **Perguntas Abertas**

**Aberta** (bom):
```
"Como você lida com retrabalho de contexto?"
"Conte-me sobre a última vez que tiveram scope creep"
```

**Fechada** (evitar):
```
"Você gasta mais de 1h em retrabalho?"
"Concorda que -50% retrabalho seria útil?"
```

### 3. **Follow-ups**

Se resposta for vaga, aprofunde:

```
Pessoa: "Cansa explicar contexto"

Você: "Cansa? Pode detalhar? Como você se sente?"
       "Qual é a frequência? 1x semana? 10x?"
       "Qual é o impacto? Tempo? Qualidade?"
```

### 4. **Validação**

Não assume, confirma:

```
Pessoa: "A gente usa Jira"

Você: "Jira pra quê? Pra rastrear tasks? Ou features também?"
      "E vocês documentam contexto estratégico lá ou em outro lugar?"
```

## Documentação Durante Entrevista

### **Formato de Notas**

```
[Timestamp] [Persona] [Pergunta #] [Resposta]

---

[5min] PM_Leader_01 Warm-up: "Lido com equipe de 8 PMs, 3 anos no role"

[10min] PM_Leader_01 P2 Retrabalho: "3-5x por semana preciso re-explicar 
prioridade. Novos membros perguntam 'por que isso?'. É exaustivo."

[15min] PM_Leader_01 P4 Push: "Frustração total. A gente entrega features
mas contexto fica perdido. 2-3 meses depois alguém pergunta 'por que fizemos isso?'"
```

### **O QUE ANOTAR**

✅ **Notar:**
- Direct quotes (entre aspas)
- Frequências ("3x/semana")
- Impactos concretos ("Perde 5h")
- Emoções (frustração, ansiedade)

❌ **NÃO anotar:**
- Sua interpretação ("Quer dizer que...")
- Opinião ("Acho que...")
- Promises ("Vou resolver isso")

## Esperado de Cada Persona

### **PM Leader**

**Validar:** VALUE_PROPOSITION resonates com problema real  
**Output esperado:**
- Push: Tempo em re-explicação contexto
- Pull: Dashboard visual Task→OKR
- Habit: Jira/Notion integrado
- Anxiety: Adoção, training

### **EM/Tech Lead**

**Validar:** Retrabalho é problema real  
**Output esperado:**
- Push: Features de-priorizadas, scope creep
- Pull: Automatic OKR validation, quality gate
- Habit: Code review workflow
- Anxiety: Integration, reliability

### **AI Architect**

**Validar:** Confiança semântica é pain real  
**Output esperado:**
- Push: Alucinação, retrabalho de context fetching
- Pull: Formal ontology, 0.0-1.0 scoring
- Habit: Existing RAG/embedding pipeline
- Anxiety: Framework integration, latency

### **Product Ops**

**Validar:** Roll-up manual é problema real  
**Output esperado:**
- Push: Hours spent in roll-up, OKR misalignment
- Pull: Automatic Task→OKR queries, dashboards
- Habit: Warehouse (Snowflake/BigQuery)
- Anxiety: Data quality, privacy

### **Early Adopter**

**Validar:** Interesse em piloto  
**Output esperado:**
- Pain + willingness to try
- Feedback loop setup
- Contract terms (free pilot, 3mo, support)
- Reference case study potential

## Troubleshooting: E Se...

### "Pessoa não responde de forma clara?"
```
Você: "Sinto que não captei bem. Posso reformular?
      Você está falando que [X], certo?
      Ou é mais [Y]?"
```

### "Pessoa discorda com VALUE_PROPOSITION?"
```
Não argumente. Curiosidade:
"Entendi, você vê de forma diferente. Por quê?
O que seria mais acurado na sua opinião?"
```

### "Pessoa não faz entrevista no horário?"
```
Não reclame. Reschedule:
"Sem problema! Qual dia funciona melhor?"
(Talvez não seja interested mesmo — ok, próxima)
```

### "Pessoa pede demo de APOS?"
```
Evite hoje:
"Ótima pergunta! Ainda estamos em fase de design.
Mas queria primeiro garantir que estamos resolvendo o problema certo.
Se você validar meu entendimento, podemos explorar depois?"
```

### "Conversa fica off-topic?"
```
Redirect gently:
"Interessante! Anotei. Voltando ao tema —
você estava falando de [X], certo?"
```
